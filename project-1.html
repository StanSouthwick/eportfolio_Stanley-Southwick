Project 01

Network Anomaly Detection

Production-grade ML pipeline for network intrusion detection — trained on 2.83M labelled flows, containerised FastAPI inference deployed on AWS ECS Fargate, with automated CI/CD and runtime drift monitoring.

Overview

This project demonstrates the full ML engineering lifecycle — from raw data exploration through to a containerised, cloud-deployed inference API with automated redeployment and runtime monitoring. It was built as a portfolio piece targeting ML engineering roles in the defence and security sector.

The dataset is CICIDS2017 (Canadian Institute for Cybersecurity), comprising 2.83 million labelled network flows across 15 traffic classes — including DoS, DDoS, PortScan, Brute Force, Web Attacks, and Infiltration. Raw CSVs are preprocessed down from 79 features to 48 by dropping zero-variance, duplicate, and high-correlation columns, producing a deterministic, reproducible feature set.

Architecture

  • Preprocessing — src/preprocess.py ingests 8 CICIDS2017 CSVs, drops redundant features, fits a StandardScaler, and exports feature_names.json and scaler.pkl to artifacts/.
  • Training — train_xgb.py trains an XGBoost classifier with sample-weight class balancing. train_anomaly.py trains an Isolation Forest exclusively on BENIGN traffic for novelty detection.
  • Evaluation & Ensemble — ensemble.py implements an IF → XGBoost gate: anomaly-flagged flows are routed to XGBoost for classification; non-flagged flows bypass IF entirely. All runs logged to MLflow.
  • Inference API — FastAPI app with four endpoints (/predict/classify, /predict/anomaly, /predict/ensemble, /predict/batch). Model artifacts pulled from AWS S3 at startup via boto3.
  • CI/CD & Drift — GitHub Actions: push to main → Docker build → ECR push → ECS Fargate redeploy. src/monitor/drift.py computes per-feature KS statistics against the training distribution, logged to MLflow.

* Ensemble macro F1 is lower than standalone XGBoost by design — Isolation Forest gates novel/unseen traffic that XGBoost was not trained on. The ensemble's value is breadth of detection, not benchmark maximisation.

Key Design Decisions

  • XGBoost with sample weights over SMOTE. SMOTE introduces synthetic samples that can misrepresent minority-class decision boundaries. Sample weights achieve the same rebalancing effect during training without modifying the data distribution — preferable for production where inference operates on raw flows.
  • No log1p transformation. Tree-based models are invariant to monotonic feature transformations. Log1p was evaluated and deliberately dropped; the notebooks record this decision with supporting evidence.
  • Isolation Forest as a supporting actor. IF is trained exclusively on BENIGN traffic to flag flows that deviate from the normal operational envelope — not to maximise labelled-benchmark F1. Hyperparameter tuning was deliberately skipped: over-fitting its contamination parameter to CICIDS2017 labels would undermine its novelty-detection purpose.
  • Preprocessing determinism. Correlation-based feature dropping uses np.triu(k=1) to ensure a stable, reproducible column set regardless of execution order or string comparison behaviour.

What I Learned

This project forced me to think beyond model accuracy. Getting XGBoost to 0.90 macro F1 was straightforward once I understood the class imbalance problem — the harder part was designing the system around it: keeping the preprocessing pipeline deterministic, giving the Isolation Forest a distinct purpose from the classifier, and understanding why the ensemble's lower benchmark number is evidence of better real-world behaviour, not worse.

Deploying to AWS ECS Fargate and wiring up GitHub Actions for CI/CD made clear how much production ML differs from notebook ML. Artifact management, environment variables, container image versioning, and drift monitoring are all things that never appear in a Kaggle competition but dominate real deployments.
